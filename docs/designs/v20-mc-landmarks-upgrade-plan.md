# PLAN: v20 Phase 3+ — MC 복셀 랜드마크 대규모 업그레이드

> 195국 랜드마크의 다양성, 텍스처, 마인크래프트 미학을 근본적으로 개선

## 1. 개요

### 현재 문제 (3가지)

**문제 1: 아키타입 중복 심각**
- MONOLITH (54국, 27.7%) + DOME (39국, 20.0%) = 93국이 단 2종
- 상위 3종(+CASTLE 18국)이 전체 195국 중 111국 (56.9%) 차지
- 아프리카 ~40국, 태평양 섬국 ~15국, 카리브 ~10국 전부 동일한 MONOLITH
- 이슬람 + 중남미 + 동유럽 등 이질적 문화가 전부 DOME

**문제 2: 텍스처 단조로움**
- 아키타입당 단색 `MeshLambertMaterial` 1개 (27종 색상 팔레트)
- MC 블록은 면(face)마다 다른 텍스처 — 현재는 전체가 밋밋한 단색
- 구조물 내부에서 블록 종류(돌, 나무, 금, 모래 등) 구분 불가
- 어떤 거리에서든 "색깔 덩어리"로만 보임

**문제 3: 마인크래프트 미학 부족**
- BoxGeometry만 사용했지만 "MC 느낌"이 안 남
- MC의 핵심: **면별 다른 텍스처** + **블록 이음새(grid line)** + **복셀 아트 색감**
- 현재는 단색 블록 덩어리 → 레고보다도 단조로움
- MC 특유의 픽셀 텍스처, 블록 그리드감, 면별 밝기 차이가 없음

### 목표
1. 아키타입 27종 → **42종+** 확장 (대륙/문화권별 고유 형태)
2. MC 스타일 **CanvasTexture 아틀라스** 도입 (면별 블록 텍스처)
3. **멀티 머티리얼** 시스템 — 하나의 건축물 내에 2-4종 블록 색상
4. 성능 목표 유지: 총 draw calls ≤ 50, 60fps

## 4. MC 텍스처 시스템 설계

### 4.1 핵심 접근: CanvasTexture 블록 아틀라스

MC의 핵심 미학은 **16×16 픽셀 텍스처**가 면(face)마다 매핑되는 것.
Three.js에서 이를 재현하는 최적 방법: **프로시저럴 CanvasTexture 아틀라스**

```
┌──────────────────────────────────────────┐
│   Block Texture Atlas (256×256 Canvas)    │
│                                           │
```

### 4.4 면별 텍스처 매핑

**MC의 핵심**: 상면/하면/측면이 다를 수 있음 (잔디 블록 = 상:초록, 측:흙+초록띠, 하:흙)

```
각 Box의 6면에 UV를 다르게 매핑:
- 기본: 6면 모두 같은 블록 텍스처 (대부분의 블록)
- 특수: 상면 다른 텍스처 (잔디, 테라스 등)
- UV 오프셋으로 아틀라스 내 서로 다른 셀 참조
```

**구현**: `BufferGeometry`의 UV attribute를 직접 수정하여 각 box face가 아틀라스의 다른 영역을 참조하도록 설정. `mergeGeometries` 전에 각 box의 UV를 블록 타입에 맞게 설정.

## 5. 멀티 머티리얼 시스템

### 5.1 기존 문제: 단색 머티리얼

현재: 아키타입당 1색 `MeshLambertMaterial` → 에펠탑이 전체 구리색 한 덩어리
MC에서 에펠탑을 만들면: 기반=돌, 다리=철 블록, 중간=구리, 꼭대기=금 블록 등 **여러 블록 타입 혼합**

### 5.2 해결: 박스별 블록 타입 지정

각 아키타입의 `createArchetypeGeometry()` 함수에서 **각 박스에 블록 타입을 지정**:

```typescript
interface VoxelBox {
  w: number; h: number; d: number;  // 크기
  x: number; y: number; z: number;  // 위치
  block: BlockType;                  // ★ 블록 타입
  topBlock?: BlockType;              // 상면 다른 블록 (선택)
}

// 예: TOWER (에펠탑)
const TOWER_VOXELS: VoxelBox[] = [
  // 다리 — 철 블록
  { w: 0.3, h: 1.2, d: 0.3, x: -0.6, y: 0.6, z: -0.6, block: BlockType.IRON },
  { w: 0.3, h: 1.2, d: 0.3, x:  0.6, y: 0.6, z: -0.6, block: BlockType.IRON },
  // 중간 플랫폼 — 구리
  { w: 1.2, h: 0.2, d: 1.2, x: 0, y: 1.3, z: 0, block: BlockType.COPPER },
  // 상부 — 금
  { w: 0.3, h: 1.0, d: 0.3, x: 0, y: 2.3, z: 0, block: BlockType.GOLD },
  // 안테나 — 다이아몬드
  { w: 0.1, h: 0.5, d: 0.1, x: 0, y: 3.3, z: 0, block: BlockType.DIAMOND },
];
```

### 5.3 UV 매핑 전략

**박스별 UV를 블록 타입에 맞게 설정** → 단일 머티리얼(아틀라스 텍스처)로 다색 구현:

1. 각 `BoxGeometry` 생성 시 UV를 아틀라스 내 해당 블록 셀로 리매핑
2. `mergeGeometries()`로 모든 박스를 하나의 BufferGeometry로 합침
3. 하나의 `MeshLambertMaterial({ map: atlasTexture })` 적용

**결과**: 아키타입당 여전히 **1 draw call** (InstancedMesh), 하지만 시각적으로 **멀티 블록 구조**

### 5.4 면별 밝기 차이 (MC 핵심)

MC의 또 다른 시각적 특징: **면 방향에 따른 밝기 차이**
- 상면: 100% 밝기 (하늘 향함)
- 동/서면: 80% 밝기
- 남/북면: 60% 밝기
- 하면: 50% 밝기

**구현**: vertex color attribute에 면 방향 기반 밝기 베이킹
```typescript
// 각 face의 vertex에 밝기 곱셈
const faceNormalBrightness = {
  top: 1.0,
  bottom: 0.5,
  north: 0.8,
  south: 0.8,
  east: 0.6,
  west: 0.6,
};
// → geometry.setAttribute('color', ...) → material.vertexColors = true
```

이 방식이면 추가 조명 계산 없이 MC 특유의 **면별 명암**이 나타남.

## 6. 성능 전략

### 6.1 현재 렌더링 구조

- 아키타입당 1개 InstancedMesh (현재 27 아키타입 = 최대 27 draw calls)
- 실제 사용 아키타입 ~20개 (backface culling으로 절반 숨김)
- LOD 3단계: Far 28국, Mid 68국, Close 195국

### 6.2 아키타입 확장 시 우려

42 아키타입 → 최대 42 draw calls? **아님.**

**핵심 인사이트**: 텍스처 아틀라스 + UV 매핑이면 **모든 아키타입이 같은 머티리얼** 공유 가능.

→ `InstancedMesh`를 아키타입별로 분리하는 이유는 **geometry가 다르기** 때문.
→ geometry가 다른 한 아키타입당 1 InstancedMesh는 불가피.
→ 하지만 머티리얼은 1개로 통합 (아틀라스 텍스처 공유)

### 6.3 최적화 전략

| 전략 | 효과 | 적용 |
|------|------|------|
| **텍스처 아틀라스 통합** | 머티리얼 1개 → 텍스처 바인딩 1회 | 전체 |
| **LOD 유지** | Far: ~15개 타입, Mid: ~25개, Close: ~42개 | GlobeLandmarks |
| **Backface culling** | 절반 인스턴스 숨김 | LandmarkMeshes |
| **Geometry 캐싱** | 아키타입당 1회 생성 | landmark-geometries |
| **Frustum culling 비활성** | 지구본 전체가 뷰에 있으므로 불필요 | InstancedMesh |
| **LOD별 geometry 간소화** | Far: 박스 수 50% 줄임 (먼 거리에서 디테일 불필요) | 선택적 |

### 6.4 예상 성능 메트릭

```
Far LOD (>300):
  - 활성 인스턴스: ~15-20국 (Tier 1)
  - Draw calls: ~12-15 (사용 중인 타입 수)
  - 삼각형: ~1,500 (평균 8 box × 12 tri × 15)

Mid LOD (160-300):
  - 활성 인스턴스: ~35-50국 (Tier 1+2)
  - Draw calls: ~20-25
  - 삼각형: ~5,000

Close LOD (<160):
  - 활성 인스턴스: ~100국 (전체 중 카메라 방향)
  - Draw calls: ~35-42
  - 삼각형: ~12,000

총합: 기존 대비 draw calls +15 증가 (27→42),
      삼각형 +4,000 (텍스처 복잡도 증가 없이 box 수만 약간 증가)
      → 60fps 유지 가능 (GPU 바운드 아닌 CPU/JS 바운드)
```

### 6.5 메모리 예산

```
텍스처 아틀라스: 256×256 RGBA = 256KB (1장)
Geometry 캐시: 42 타입 × 평균 1KB = 42KB
InstancedMesh matrix: 195 × 64byte = 12.5KB per type
총 추가 메모리: ~350KB (현재 ~200KB → 대비 +150KB)
```
