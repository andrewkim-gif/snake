# ADR-029: R3F + InstancedMesh 렌더링 전략

## Status
Accepted

## Context
V38 Canvas 2D → Three.js 3D 변환에서 200+ 엔티티(적, 투사체, 파티클)를 60fps로 렌더링해야 합니다. 각 엔티티를 개별 Mesh로 만들면 draw call이 수백 개에 달해 GPU 병목이 됩니다.

## Decision
- **InstancedMesh-first 전략**: 같은 geometry를 공유하는 엔티티는 InstancedMesh로 batch 렌더링합니다.
- 적: template별 InstancedMesh (10-15 pools, 각 capacity 100)
- 투사체: 카테고리별 InstancedMesh (10 pools, 각 capacity 50-200)
- 파티클: 단일 InstancedMesh (capacity 500)
- **BatchedMesh는 선택적 최적화**: S23에서 A/B 벤치마크 후 결정. 실험적 API이므로 InstancedMesh fallback 유지.
- per-instance color는 `setColorAt()` API 사용
- LOD LOW는 모든 적을 단일 colored cube InstancedMesh로 통합

## Consequences
- **이점**: draw call 200+ → 35-60으로 감소, 60fps 달성 가능
- **제약**: 같은 pool 내 엔티티는 동일 geometry + material 필수
- **복잡도**: pool 관리(spawn/despawn), instance index tracking 필요
- BatchedMesh 불안정 시 InstancedMesh만으로도 목표 달성 가능

## Alternatives Considered
1. **Individual Mesh**: 단순하지만 draw call 폭발 (200적 = 200 calls). 제외.
2. **BatchedMesh only**: 서로 다른 geometry를 1 draw call로 통합 가능하지만, Three.js r163+ 실험적 API. per-instance color 지원 제한. 1순위 대안으로 보류.
3. **Points/Sprites**: 경량이지만 3D 깊이감 부족. Voxel 미학과 불일치. 제외.
