# ADR-024: Layered Error Propagation Strategy

## Status

Accepted

## Context

A real-time game server handles errors differently from traditional web applications. Errors in a 20Hz game loop must not block the tick, crash the room, or leak internal state to clients. We need a consistent error handling strategy across all layers.

Options:
1. **Fail-fast everywhere**: Any error propagates up and crashes the component
2. **Catch-all recovery**: Every goroutine has `recover()`, errors are logged and swallowed
3. **Layered propagation**: Errors classified by severity, each layer has defined behavior
4. **Error channels**: Errors sent to a central error handler via Go channels

## Decision

**Option 3: Layered error propagation.**

Four error layers with distinct behaviors:

### Layer 1 -- Game Logic Errors (Non-Fatal)
```
Examples: invalid upgrade choice, out-of-bounds agent, expired action timer
Behavior: log at DEBUG, silently correct or ignore
Rationale: server-authoritative model means these are client misbehavior, not bugs
```

### Layer 2 -- Protocol Errors (Client-Facing)
```
Examples: unknown event name, malformed JSON, rate limit exceeded
Behavior: log at INFO, send {e:"error", d:{code, message}} to offending client
Codes:    4001=bad_event, 4002=malformed_payload, 4003=rate_limited, 4004=not_in_room
Rationale: client needs feedback for debugging; info-level helps track abuse patterns
```

### Layer 3 -- Connection Errors (Connection-Fatal)
```
Examples: ReadLimit exceeded, send buffer full, ping timeout
Behavior: log at WARN, close WebSocket connection gracefully
Rationale: connection is unrecoverable, client will auto-reconnect
```

### Layer 4 -- System Errors (Component-Fatal)
```
Examples: Room goroutine panic, JSON marshal failure, unexpected nil pointer
Behavior: log at ERROR, recover() in goroutine, restart component
Rationale: must not crash the process; isolated recovery per component
```

## Consequences

**Positive:**
- Clear contract for each error type: developers know how to handle new errors
- No error can crash the entire server (Layer 4 recovery)
- Minimal client-visible errors (Layer 1 is invisible, Layer 2 is informative)
- Log levels match severity (DEBUG->INFO->WARN->ERROR) for filtering
- Game loop never blocks on error handling

**Negative:**
- Layer 1 errors could mask bugs (silent correction hides root cause)
- Layer 4 `recover()` can mask serious issues if too aggressive
- Error code catalog (Layer 2) needs maintenance as protocol grows

**Mitigation:**
- Development builds use `LOG_LEVEL=debug` to surface Layer 1 errors
- Layer 4 recovery always logs full stack trace via `slog.Error`
- Error codes documented in protocol spec (Section 7)

## Alternatives Considered

- **Option 1 (fail-fast)**: Rejected -- a single malformed client message would crash the Room, affecting 20+ players.
- **Option 2 (catch-all)**: Rejected -- swallowing all errors makes debugging impossible. Layering provides precision.
- **Option 4 (error channels)**: Over-engineered -- adds channel overhead to the hot path (20Hz tick). Direct error handling within each layer is simpler and faster.
