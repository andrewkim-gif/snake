# ADR-019: Upgrade System -- Server-Side Timeout with Random Fallback

## Status

Accepted

## Context

When an agent levels up, the server presents 3 upgrade choices via the `level_up` event. The player (or AI agent) must select one within 5 seconds (100 ticks at 20Hz). Several design options exist for handling timeout:

1. **No timeout**: Player can delay indefinitely (blocks gameplay, exploitable)
2. **Client-side timeout**: Client auto-selects after 5s (bypassable, non-authoritative)
3. **Server-side timeout with random fallback**: Server tracks deadline tick; if no `choose_upgrade` received, server randomly selects
4. **Server-side timeout with first choice**: Always pick choice index 0

## Decision

**Option 3: Server-side timeout with random fallback.**

The server sets `agent.UpgradeDeadline = currentTick + 100` when generating choices. On each tick, `UpgradeSystem.ProcessTimeouts()` checks if the deadline has passed. If so, it randomly selects one of the 3 choices and applies it.

```go
func (u *UpgradeSystem) ProcessTimeouts(agents map[string]*Agent, tick uint64) {
    for _, agent := range agents {
        if agent.PendingChoices != nil && tick >= agent.UpgradeDeadline {
            choice := rand.Intn(len(agent.PendingChoices))
            u.ApplyUpgrade(agent, choice)
            agent.PendingChoices = nil
        }
    }
}
```

## Consequences

**Positive:**
- Server-authoritative: no client-side bypass possible
- Fair to AI agents: 5 seconds is ample time for LLM API call
- Random fallback prevents exploiting specific default choices
- Game flow never stalls waiting for player input

**Negative:**
- Players who genuinely need more time (mobile, accessibility) get random upgrades
- Slight complexity in handling `choose_upgrade` arriving at the exact deadline tick

**Edge Case -- Race Condition:**
If `choose_upgrade` arrives at tick 99 and timeout fires at tick 100:
- Room goroutine processes `inputChan` in `select` alongside `gameTicker.C`
- Both cases execute in the same goroutine (no concurrency issue)
- If `choose_upgrade` is in the channel when timeout fires, the `inputChan` case may process first (Go select is random among ready cases)
- Mitigation: `ApplyUpgrade` checks `PendingChoices != nil` before applying; double-apply is impossible

## Alternatives Considered

- **Option 1 (no timeout)**: Rejected -- players could grief by never choosing
- **Option 2 (client timeout)**: Rejected -- violates server-authoritative principle
- **Option 4 (first choice)**: Rejected -- predictable fallback could be gamed
