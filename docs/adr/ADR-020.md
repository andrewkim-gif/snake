# ADR-020: No Session Resume on Reconnection

## Status

Accepted

## Context

When a WebSocket connection drops (network glitch, tab switch on mobile, ISP hiccup), the client auto-reconnects via exponential backoff. The question is whether to resume the player's existing agent or treat it as a fresh join.

Options:
1. **Session resume**: Server "parks" disconnected agent for N seconds, client resends session token to reclaim
2. **Fresh join**: Disconnection = agent removed, reconnection = new agent spawned with grace period
3. **Spectator on reconnect**: Player can watch their old room but must wait for next round to play

## Decision

**Option 2: Fresh join (no session resume).**

On disconnect:
1. Hub.unregister removes client from room's client map
2. Room.leaveChan receives clientID
3. Agent is removed from arena (death orbs scattered if alive)
4. No server-side session storage

On reconnect:
1. Client treated as brand new connection
2. Must send `join_room` again
3. New agent spawned with 30-second grace period

## Consequences

**Positive:**
- Dramatically simpler server architecture (no session storage, no timeout management, no "parked agent" state)
- No ambiguity about agent ownership
- Works identically for humans and AI agents
- Grace period provides fair protection for reconnected players
- 5-minute rounds limit the cost of losing progress

**Negative:**
- Player loses their build/level/position on any disconnect
- Players on unstable connections may have a poor experience
- Competitive players may feel frustrated losing a strong build

**Mitigating Factor:**
The game's core loop (5-minute rounds, frequent resets) means the maximum loss is one round's worth of progress. Unlike persistent MMOs, there's no long-term state to recover.

## Alternatives Considered

- **Option 1 (session resume)**: Rejected -- requires session tokens, server-side agent parking, timeout cleanup, ownership verification. Complexity budget is better spent on game features. Can revisit in Phase 5 if retention data shows disconnect-related churn.
- **Option 3 (spectator)**: Rejected -- adds a "spectator" mode feature just for reconnection; over-engineered for the use case.
