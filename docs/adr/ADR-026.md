# ADR-026: CountryArena Embeds Room (Composition)

## Status
Accepted

## Context
v10 has a clean `Room` struct that manages the state machine (waiting → countdown → playing → ending → cooldown) and owns an `Arena` for the actual game loop. v11 needs to add country-specific context (tier, terrain, resources, faction scoring) without breaking the battle core.

## Decision
Use **Go struct embedding** (composition) to create `CountryArena` that wraps `Room`:

```go
type CountryArena struct {
    *Room                    // Embed v10 Room (state machine + Arena)
    CountryISO  string       // ISO 3166-1 alpha-3
    CountryData *CountryData // Tier, resources, terrain, neighbors
    FactionScores map[string]int // Faction ID → aggregate score
    BattleType  BattleType   // arena, skirmish, siege, etc.
    DefenseBuff float64      // From blockchain oracle (0.0 - 0.30)
}
```

This preserves all Room behavior (state transitions, broadcast, player management) while adding:
- Faction-aware score aggregation after battle
- Terrain theme selection based on country
- Defense buff application (defender advantage + blockchain oracle)
- Sovereignty transition logic (20% threshold)
- Battle result emission to Redis

## Consequences
**Easier:**
- Zero changes to Room or Arena code (v10 game loop preserved)
- All existing tests continue to pass
- Clear separation: Room handles battle lifecycle, CountryArena handles geopolitics

**Harder:**
- Some methods need thin wrappers (e.g., `endRound()` must also aggregate faction scores)
- Two levels of abstraction to understand for new developers

## Alternatives Considered
1. **Modify Room directly**: Rejected — breaks v10 compatibility, mixes concerns
2. **Interface-based approach**: Rejected — Room has too many methods to wrap as interface
3. **Fork Room into CountryRoom**: Rejected — code duplication, maintenance burden
