# ADR-023: Feature Flags via Environment Variables

## Status

Accepted

## Context

Agent Survivor is developed in phases (S01-S59). Features like MapObjects, Synergies, Agent API, Coach, and Progression are added incrementally. We need a mechanism to enable/disable features at deployment time without code changes.

Options:
1. **Build-time flags**: Go build tags (`-tags=synergies`) compile different binaries
2. **Environment variables**: Runtime `ENABLE_SYNERGIES=true` checks
3. **External feature flag service**: LaunchDarkly, Unleash, or custom Redis-based
4. **Config file**: YAML/JSON config loaded at startup

## Decision

**Option 2: Environment variables.**

```go
type Config struct {
    // ... existing fields
    EnableMapObjects  bool `envconfig:"ENABLE_MAP_OBJECTS" default:"false"`
    EnableSynergies   bool `envconfig:"ENABLE_SYNERGIES" default:"false"`
    EnableAgentAPI    bool `envconfig:"ENABLE_AGENT_API" default:"false"`
    EnableCoach       bool `envconfig:"ENABLE_COACH" default:"false"`
    EnableProgression bool `envconfig:"ENABLE_PROGRESSION" default:"false"`
}

// Usage in game loop:
if a.config.EnableSynergies {
    a.upgrade.CheckSynergies(agent.Build)
}
```

New features default to `false` and are enabled explicitly in Railway environment variables when ready for production.

## Consequences

**Positive:**
- Zero additional dependencies (no external service, no config file parsing)
- Railway-native: environment variables are first-class in Railway dashboard
- Instant toggling: change env var + restart container (~30s)
- Consistent with existing `PORT`, `CORS_ORIGIN` pattern
- `envconfig` already in dependency tree

**Negative:**
- No gradual rollout (no percentage-based flags)
- No per-user targeting (all-or-nothing)
- Requires container restart to take effect (acceptable for game server)
- Flag proliferation risk if many features added

**Why acceptable:**
- Phase 1-3 are primarily additive features, not A/B tests
- "All-or-nothing" toggling matches the phase-based development model
- Container restart is fast (~30s) and game rounds are transient
- When per-user targeting is needed (Phase 5+), migrate to a proper feature flag service

## Alternatives Considered

- **Option 1 (build tags)**: Rejected -- requires separate Docker images per feature combination. Deployment complexity scales exponentially.
- **Option 3 (external service)**: Rejected -- adds network dependency, cost, and complexity. Unnecessary for 5-10 boolean flags.
- **Option 4 (config file)**: Viable but adds file management. Environment variables are simpler and Railway-native.
