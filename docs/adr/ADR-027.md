# ADR-027: On-Demand Arena Creation with Object Pool

## Status
Accepted

## Context
195 countries cannot all have active arenas simultaneously. At ~5MB per arena (agents, orbs, spatial hash, collision data), 195 concurrent arenas would require ~1GB just for game state. Phase 1 targets 500 CCU with ~20 concurrent battles.

## Decision
Implement an **arena object pool** with on-demand allocation:

- **Pool size**: Max 50 concurrent arenas (configurable)
- **Pre-warm**: 10 arenas pre-allocated at startup
- **Allocation**: Tier-based sizing (S=6000px, D=1500px arena dimensions)
- **Creation trigger**: Country has ≥3 deployed agents AND battle cycle starts
- **Release**: After battle cooldown (15s), reset and return to pool
- **Eviction**: If idle for 10 min, deallocate from pool

For countries with 0 deployed agents (most of the 195), no arena is created. A simple NPC defender presence is tracked in Redis without a full arena instance.

## Consequences
**Easier:**
- Memory usage predictable and bounded (~250MB for 50 active arenas)
- Arena recycling avoids GC pressure from frequent allocation
- Clear capacity planning (50 arenas = known memory ceiling)

**Harder:**
- Pool exhaustion requires graceful degradation (queue or reject new battles)
- Arena reset must be thorough (no state leaks between battles)

## Alternatives Considered
1. **Always-on arenas for all 195**: Rejected — ~1GB minimum, most idle
2. **Lazy creation without pool**: Rejected — GC pressure from frequent alloc/dealloc
3. **Kubernetes HPA per arena**: Rejected — over-engineered for Phase 1
