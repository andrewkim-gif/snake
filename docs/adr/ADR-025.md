# ADR-025: Single Binary with Internal Packages (Phase 1)

## Status
Accepted

## Context
v11 introduces two logical servers (Game Server for real-time combat, Meta Server for REST API and meta-game logic). We need to decide whether to deploy them as separate services or a single binary.

At Phase 1 (MVP), we expect ~500 CCU and ~20 concurrent battles. Splitting into microservices at this stage adds network latency, operational complexity, and deployment coordination overhead without commensurate benefits.

## Decision
Deploy as a **single Go binary** with clear internal package separation:
- `server/internal/game/` — Game server logic (arenas, combat, agents)
- `server/internal/world/` — World management (WorldManager, CountryArena)
- `server/internal/meta/` — Meta-game logic (factions, economy, diplomacy)
- `server/internal/api/` — REST API handlers
- `server/internal/auth/` — Authentication
- `server/internal/ws/` — WebSocket hubs
- `server/internal/db/` — PostgreSQL access
- `server/internal/cache/` — Redis access

Communication between Game and Meta is via:
1. Go channels (in-process, zero latency)
2. Redis pub/sub (for events that need persistence/fanout)

## Consequences
**Easier:**
- Single deployment, single health check, single log stream
- Zero network latency between Game and Meta
- Simpler local development (one `go run`)

**Harder:**
- Cannot scale Game and Meta independently (addressed in Phase 2)
- Single point of failure (mitigated by Railway auto-restart)

## Alternatives Considered
1. **Two separate services from day 1**: Rejected — premature complexity for MVP scale
2. **Serverless functions for Meta API**: Rejected — WebSocket support is critical
3. **gRPC between services**: Rejected — unnecessary overhead at single-binary scale

## Migration Path
Phase 2: Extract Meta Server into separate binary; replace Go channels with Redis pub/sub (already in use). The internal package structure makes this a mechanical refactoring.
