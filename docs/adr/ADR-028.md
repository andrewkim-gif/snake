# ADR-028: Redis Pub/Sub for Game-Meta Communication

## Status
Accepted

## Context
The Game Server (real-time combat) and Meta Server (REST API, persistence) need to exchange data:
- Battle results (Game → Meta): triggers sovereignty updates, economy effects
- Deploy/recall commands (Meta → Game): spawns/removes agents in arenas
- Sovereignty updates (Meta → Game): applies defense buffs in next battle

In Phase 1 these are in the same binary; in Phase 2+ they may be separate services.

## Decision
Use **Redis pub/sub** as the primary inter-module communication mechanism:

Channels:
- `battle:{iso}` — Battle results from Game to Meta
- `deploy:{iso}` — Agent deployment commands from Meta to Game
- `sovereignty:{iso}` — Sovereignty changes from Meta to Game/Client
- `global:events` — World events from Workers to all
- `economy:tick` — Economy updates from Workers to Meta
- `blockchain:buyback` — Buyback triggers from Meta to BlockchainWorker

In Phase 1, both publisher and subscriber are in the same process, but still go through Redis. This ensures:
1. The communication pattern is already correct for Phase 2 splitting
2. Messages are observable (Redis MONITOR for debugging)
3. Worker processes can subscribe independently

## Consequences
**Easier:**
- Zero-effort service splitting in Phase 2 (just deploy separate binaries)
- Natural event log (Redis pub/sub can be monitored)
- Workers can subscribe to events without direct coupling

**Harder:**
- Redis becomes a critical dependency (mitigated by health checks + auto-restart)
- 1-2ms added latency vs Go channels (acceptable for non-tick-critical messages)
- No message persistence (pub/sub is fire-and-forget — critical results also write to PostgreSQL)

## Alternatives Considered
1. **Go channels only (Phase 1) + refactor later**: Rejected — refactoring pub/sub pattern later is expensive
2. **NATS/Kafka message queue**: Rejected — over-engineered for Phase 1 scale
3. **HTTP callbacks between services**: Rejected — synchronous coupling, retry complexity
